> 版本: v1.0

## 一、项目背景

### 1.1 现状问题

- 开源用户无法自助安装企业插件（需人工配置 CRD 和访问地址）
- 授权流程繁琐（导出集群信息 → 人工生成授权码 → 填入平台）
- 插件无版本管理，无法自助升级
- 无法区分开源插件和企业插件

### 1.2 目标

让用户能在插件中心自助完成：
- 查看可用插件列表（区分开源/企业）
- 填入授权码激活企业功能
- 一键安装和升级插件

## 四、授权机制设计

### 4.1 授权码结构

授权码包含以下信息（加密）：
- 授权类型（基础版 / 高级版）
- 绑定的 Cluster UID
- 有效期（开始时间、结束时间）
- 可安装的插件列表（可选，用于精细控制）

### 4.2 Cluster UID 获取

使用 kube-system namespace 的 UID 作为集群唯一标识：

```bash
kubectl get namespace kube-system -o jsonpath='{.metadata.uid}'
```

### 4.3 授权验证流程

```
用户填入授权码
      │
      ▼
解密授权码，提取信息
      │
      ▼
获取当前集群 UID
      │
      ▼
比对 Cluster UID ──── 不匹配 ──▶ 提示"授权码与当前集群不匹配"
      │
      │ 匹配
      ▼
检查有效期 ──── 已过期 ──▶ 提示"授权已过期"
      │
      │ 有效
      ▼
保存授权信息，激活成功
```

### 4.4 插件下载授权

插件包从远程仓库下载时需要验证：
- 请求头携带授权信息
- 服务端验证授权有效性和版本权限
- 验证通过后返回下载地址或直接返回插件包

## 五、打包机制

### 5.1 插件包格式

使用现有 RAM 格式，包含：
- 容器镜像
- 应用模板元数据
- 配置信息
- 插件描述信息

### 5.2 插件元数据

每个插件需要包含以下元数据：

```yaml
name: rainbond-enterprise-base
display_name: 企业基础插件
description: 包含应用备份、权限管理、个性化设置等企业功能
version: 1.3.0
tier: basic  # basic | advanced | opensource
icon: https://xxx/icon.png
doc_url: https://www.rainbond.com/docs/plugins/enterprise-base
changelog: |
  - 新增功能 A
  - 修复问题 B
```

### 5.3 版本管理

- 每个插件有独立的版本号
- 支持查询最新版本
- 支持版本比对，判断是否可升级

## 六、页面交互规范

### 6.1 插件卡片状态

| 状态 | 样式 | 可用操作 |
|------|------|----------|
| 未安装 + 可安装 | 正常样式 | 文档、安装 |
| 未安装 + 无权限 | 置灰 + 锁图标 | 文档、查看授权要求 |
| 已安装 | 正常样式 + 已安装标签 | 文档、卸载 |
| 已安装 + 可升级 | 正常样式 + 升级提示 | 文档、升级、卸载 |
| 安装中 | 加载状态 | 无 |

### 6.2 错误处理

| 场景 | 提示内容 |
|------|----------|
| 授权码格式错误 | "授权码格式不正确，请检查后重试" |
| 授权码与集群不匹配 | "该授权码与当前集群不匹配" |
| 授权已过期 | "授权已过期，请联系销售续费" |
| 下载失败 | "插件下载失败，请检查网络后重试" |
| 安装失败 | "插件安装失败：{错误信息}" |

## 七、非功能需求

### 7.1 性能要求

- 插件列表加载时间 < 2 秒
- 授权验证时间 < 1 秒

### 7.2 兼容性

- 支持离线环境（插件包可提前下载）
- 与现有授权机制兼容

### 7.3 安全性

- 授权码加密存储
- 插件下载需验证授权
- 防止授权码在不同集群复用

## 八、验收标准

- [ ] 插件中心页面展示插件列表（区分开源/基础版/高级版）
- [ ] 可填入授权码激活企业功能
- [ ] 授权后可一键安装企业插件

- [ ] 授权过期后平台功能受限
