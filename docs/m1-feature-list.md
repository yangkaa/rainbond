## 三、技术实现要点

### 3.1 授权信息结构

```go
type LicenseToken struct {
    // 基础信息
    ClusterID      string   `json:"cluster_id"`       // 集群唯一标识
    Tier           string   `json:"tier"`             // basic / advanced

    // 时间控制
    ExpireAt       int64    `json:"expire_at"`        // 授权到期时间
    SubscribeUntil int64    `json:"subscribe_until"`  // 订阅到期时间

    // 权限控制
    AllowedPlugins []string `json:"allowed_plugins"`  // 允许安装的插件列表

    // 资源限制
    NodeLimit      int      `json:"node_limit"`       // 节点数限制
    MemoryLimit    int64    `json:"memory_limit"`     // 内存限制 (bytes)
    CPULimit       int64    `json:"cpu_limit"`        // CPU 限制 (millicores)

    // 签名 (RSA 签名)
    Signature      string   `json:"signature"`
}
```

### 3.2 存储位置

| 数据 | 存储位置 | 说明 |
|------|----------|------|
| 授权信息 | ConfigMap: `rbd-system/rbd-license-info` | 带签名，其他插件可读取验证 |
| AccessKey | Secret: `rbd-system/rbd-license-secret` | 加密存储 |
| 公钥 | 编译内置 | 每个企业插件二进制内置 |
| 私钥 | 你们的授权服务 | 不在用户环境 |

### 3.3 关键流程

**授权验证流程：**
```
用户输入授权码
    │
    ▼
授权插件解析授权码
    │
    ▼
用内置公钥验证签名 ──── 失败 ──▶ 返回"授权码无效"
    │
    │ 成功
    ▼
检查 cluster_id ──── 不匹配 ──▶ 返回"集群不匹配"
    │
    │ 匹配
    ▼
检查 expire_at ──── 已过期 ──▶ 返回"授权已过期"
    │
    │ 有效
    ▼
写入 ConfigMap + 存储 AccessKey
    │
    ▼
返回成功
```

**插件启动验证流程：**
```
企业插件启动
    │
    ▼
读取 ConfigMap: rbd-license-info
    │
    ├── 不存在 ──▶ 拒绝启动/降级模式
    │
    ▼
用内置公钥验证签名 ──── 失败 ──▶ 拒绝启动
    │
    │ 成功
    ▼
检查 cluster_id ──── 不匹配 ──▶ 拒绝启动
    │
    ▼
检查 allowed_plugins 包含自己 ──── 不包含 ──▶ 拒绝启动
    │
    ▼
检查 expire_at ──── 已过期 ──▶ 拒绝启动
    │
    ▼
正常启动，开始定时重验证
```

## 五、依赖关系

```
应用市场 (AccessKey 验证)
    │
    ▼
企业授权插件 (必须先安装)
    │
    ├── 提供授权验证服务
    ├── 管理 AccessKey
    └── 写入授权信息到 ConfigMap
    │
    ▼
其他企业插件 (依赖授权插件)
    │
    ├── 读取 ConfigMap 验证授权
    ├── 用内置公钥验证签名
    └── 定时重验证
```
