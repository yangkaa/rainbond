┌─────────────────────────────────────────────────────────────────────────────┐
│                      改进后的授权验证架构                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │

│  │  }                                                                  │   │
│  └─────────────────────────────────────────────────────────────────────┘   │

│  │  2. 用内置公钥验证签名 → 签名无效则拒绝启动                           │   │

│  │  4. 检查 expire_at 是否过期                                          │   │

│                                                                             │
│  攻击者无法绕过:                                                             │
│  ├── 伪造授权服务? → 没有私钥，无法生成有效签名                              │
│  ├── 修改授权信息? → 签名验证失败                                           │
│  ├── 用 A 插件授权运行 B? → Plugin ID 不在 allowed_plugins 中               │
│  └── 复制到其他集群? → cluster_id 不匹配                                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

前端 JS 保护方案改进

问题：前端 JS 没法编译内置公钥，容易被修改

方案：前端资源打包到后端容器，由后端服务提供

│  │                                                                     │   │

│  │  ├── 内置公钥 + 验证逻辑                                             │   │

│  │  │                                                                  │   │
│  │  └── HTTP 端点:                                                      │   │
│  │      ├── GET /static/main.js  → 先验证授权，通过才返回               │   │

│  │                                                                     │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │

│                              └── 失败 → 返回 403                            │
│                                                                             │
│  优势:                                                                       │
│  ├── 前端 JS 无法被单独替换 (嵌入二进制)                                     │
│  ├── 授权验证在插件自身，不依赖授权插件是否在线                               │
│  └── 签名验证保证授权信息真实性                                              │
│                                                                             │

但更好的方案：授权信息存储在固定位置的 ConfigMap

授权插件安装后:
1. 用户输入授权码
2. 授权插件验证并解密
3. 将授权信息(带签名)写入固定 ConfigMap: rbd-system/rbd-license-info
4. 其他插件启动时直接读取这个 ConfigMap
5. 用内置公钥验证签名

这样其他插件完全不依赖授权插件是否在线，只需要读取 ConfigMap。

授权信息结构设计

// 授权信息结构 (JWT-like)
type LicenseToken struct {
// Header
Algorithm string `json:"alg"` // "RS256"

      // Payload
      ClusterID      string   `json:"cluster_id"`
      Tier           string   `json:"tier"`           // basic, advanced
      ExpireAt       int64    `json:"expire_at"`      // Unix timestamp
      SubscribeUntil int64    `json:"subscribe_until"` // 订阅期限（用于更新）
      AllowedPlugins []string `json:"allowed_plugins"`
      NodeLimit      int      `json:"node_limit"`
      MemoryLimit    int64    `json:"memory_limit"`

      // Signature (用你们的私钥签名)
      Signature string `json:"signature"`
}

// 每个插件内置的验证函数
func VerifyLicense(token *LicenseToken, pluginID string) error {
// 1. 验证签名
if !verifySignature(token, embeddedPublicKey) {
return errors.New("invalid signature")
}

      // 2. 检查集群 ID
      currentClusterID := getClusterUID()
      if token.ClusterID != currentClusterID {
          return errors.New("cluster id mismatch")
      }

      // 3. 检查过期
      if time.Now().Unix() > token.ExpireAt {
          return errors.New("license expired")
      }

      // 4. 检查插件是否被授权
      if !contains(token.AllowedPlugins, pluginID) {
          return errors.New("plugin not authorized")
      }

      return nil
}
